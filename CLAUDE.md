# プロジェクト概要
Reactで構築されたWebサービスのフロントエンド

# 主要コマンド
- npm run dev: 開発サーバー起動
- npm run test: テスト実行

# 新機能追加時
1. 調査
2. 計画
3. 実装
4. テスト

# バグ修正時
1. 原因調査
2. テスト作成（TDD）
3. 修正実装
4. 動作確認

# コーディング規約
- TypeScriptの厳密モードを使用
- コンポーネントはfunction宣言で定義
- テストはVitestで記述
- if文は1行の処理でも中括弧`{ }`を省略しない。処理は必ず改行して記述する（`{ return; }`のように1行に収めない）
- マジックナンバーを直接記述しない。必ず名前付き定数に切り出す（定数の値は意図が読み取れる形で書く）
- メソッド名にパラメータ情報（`ById`・`ByName`等）を含めない（例: `findVideo(id)`であり`findVideoById(id)`としない）。ただし複数パラメータで識別が必要な場合は許容する
- コメントには「何をするか」のみを書く。「どこから取得するか・どのように実装するか」という実装詳細は書かない

## 指示・回答について
- 回答は日本語で行うこと
- 指示をする側も人間で間違うことがある。こちらの指示を批判的に見て、間違っていることは指摘すること
- 指示された内容についてそのまま鵜呑みにせず、背景や説明不足だと感じたら確認をとること
- 指示された案以外にも他に案がないかも考えて、良さそうな案があれば提案すること
- すべての応答を生成する前に、`.claude/skills/*/SKILL.md` を Glob してプロジェクト内の利用可能なスキル一覧を取得し、ユーザーの発話・状況と照合すること。合致するスキルがある場合、Skill ツールの呼び出しを最初のアクションとする
  - Glob を実行する際は必ず `path` パラメータにリポジトリルートを指定すること（作業ディレクトリがサブフォルダの場合、path 未指定では `.claude/` が見つからない）
  - Glob の結果が空でも「存在しない」と即断しないこと。ファイルパスが既知の場合は Read で存在を二重確認すること
- スキルを実行する前に `.claude/incidents.md` を Read し、実行するスキル名タグ（例: `#feature-impl`）のインシデントを確認すること。ファイルが存在しない場合は `.claude/incidents-template.md` を元に作成してから続行すること
- ユーザーからの修正・指摘発話（typo・細かい文言調整を除く）を検知したとき、次のように確認すること：「今回の指摘を #[スキル名] のインシデントとして記録しますか？（記録内容: [指摘の要約を1行で]）」
- 既存の skill / agent を修正・流用する提案をする場合は、必ず事前にその内容を Read で確認してから判断すること
- 設計・実装の方針を提案する前に、プロジェクト内の類似パターンや既存の実装を確認してから判断すること

## 設計指針
- 実装する時は、初めからコードを書き始めるのではなく、まずはどのような設計で実装するかを提案すること
- 考える時は次の指針に従うこと
 - いろんな軸から複数案を考え、その案のメリットデメリットを併記すること
 - その複数案から、今達成したいことを実現するシンプルかつメンテナンス性に優れた案を1つ選び提案すること
 - 選んだ案を採用する時、未来でどういう変更が加わったらその案が破綻するかも一緒に提案すること

## コードレビュー指針
- コードレビュー依頼時はあなたに調査してもらいつつ、その妥当性を私も一緒にチェックしたい
- コード調査をするときは結論だけを報告するのではなく、調査プロセスも報告すること
- 結論をまとめるときは、結論に対する対する重要な証拠も一緒に提示すること
- 最新のファイル内容を前提にレビューすること

## 実装指針
- 段階的に進める: 一度に全てを変更せず、小さな変更を積み重ねる
- 複数のタスクを同時並行で進めない
- エラーは解決してから次へ進む
- エラーを無視して次のステップに進まない
- 指示にない機能を勝手に追加しない

## エラー対応指針
- 必ず「なぜ期待とは違う現象が起きたか」について深掘りしてから次の対処を考えること

# 注意事項
- 環境変数は .env で管理（コミット禁止）
- .env / *.env / .env.* の参照・出力・実行を禁止
- 秘密情報（APIキー、トークン、認証情報）を推測しない
- 明示的に指示した場合を除き、ファイルの作成・修正・削除を行わない
- ファイル作成時に、そのファイルがGithubに挙げられるべきではないと判断した場合には、必ず.gitignoreに指定すること
- 修正を行う際には必ず以下を順守すること
  - 該当修正によって他の処理に問題がないか慎重に確認を行って作業を行うこと
  - 他の動作に関しても修正が必要な場合は既存の期待値の動作が正常に起動するように修正すること
  - 修正内容をコミットを実施しないこと
