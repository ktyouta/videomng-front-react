# Expected Output — インシデント記録

## ケース1: 口語的な指摘表現の変換

### 入力状況
ユーザーが並び替えリストのラベル削除に伴うレイアウト崩れについて、
「普通距離を近づけるでしょ」と口語的に指摘した。記録に同意した。

### NG（そのまま引用してしまう例）

```
## #ui-consistency-review 2026-07-19
内容: 見出しと説明文の間隔について、ユーザーから「普通距離を近づけるでしょ」と指摘された。
```

### OK（客観的に要約する）

```
## #ui-consistency-review 2026-07-19
内容: 見出しと説明文のように近接すべき要素同士の間隔が離れすぎている点を指摘された。
再発防止策: flexboxのgapを使うコンテナに新規要素を追加する際、近接させたい要素同士とセクション間隔を区別し、区別が必要な場合は入れ子コンテナでグループ化する。
```

## ケース2: 設計批判の変換

### 入力状況
ユーザーが共通コンポーネントの設計について、
「なんで共通なのに決め打ちで外部に影響を与える要因を持ってるの？」と批判した。記録に同意した。

### NG（そのまま引用してしまう例）

```
## #design-proposal 2026-07-23
内容: ユーザーから「なんで共通なのに決め打ちで外部に影響を与える要因を持ってるの？」と指摘を受けた。
```

### OK（客観的に要約する）

```
## #design-proposal 2026-07-23
内容: 共通コンポーネントが自分自身の外部余白（margin）を決め打ちで持っている設計を前提に対症療法的な案を提示したが、共通部品が外部レイアウトに影響する値を自分で決め打ちして持つべきではないという指摘を受けた。
再発防止策: 複数案を洗い出す際、既存コードの前提自体が問題の原因になっていないかをStep1で先に確認し、前提を解消する案も候補に含める。
```

## ケース3: 感情的な発言の変換

### 入力状況
ユーザーが検証不足を怒りとともに指摘した。「してないと思うけど？お前どこ見てる？」という発言があった。記録に同意した。

### NG（そのまま引用してしまう例）

```
内容: ユーザーから「してないと思うけど？お前どこ見てる？」と強く指摘された。
```

### OK（客観的に要約する）

```
内容: 検証不足を指摘された。
```

## 共通ルール

- 変換後の文章に鍵括弧「」で始まる箇所がある場合、その中身がユーザーの発言そのものになっていないか必ず見直す
- 感情表現（強い言葉・怒りの表明）を「客観的な事実」に薄めて書く。感情の強さそのものを記録する必要はない
