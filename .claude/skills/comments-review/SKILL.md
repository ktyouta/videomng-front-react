# Comments Review Skill

## Overview

実装後に、コメントの品質に関する違反パターンをチェックする。

---

## Check Instructions

### WHAT コメント禁止
- コードが「何をするか（WHAT）」を説明するコメントを書いていないか
- 関数名・変数名から明らかにわかる内容をコメントで説明していないか
- ※ JSDoc コメント（`/** ... */` 形式）はメソッド・関数のドキュメントとして許可（禁止対象はインラインの WHAT コメントのみ）
- ローカル変数・state 系変数（useState / useRef / useMemo / hooks 返り値）は以下の基準で判断する
  - 名前から意図が明確 かつ 定義直後にしか使わない → コメント不要
  - 名前だけでは役割が伝わりにくい、または複数箇所で参照される → WHAT コメントを付ける

### WHY コメントのみ許可
- コメントは「なぜそうするか（WHY）」に限定されているか
- 隠れた制約・非自明な不変条件・特定バグへの回避策など、読者が驚くような内容のみコメントすべき

---

## Procedure

1. 変更されたファイルのコメントを確認する
2. 各コメントが WHAT か WHY かを判断する
3. WHAT コメントを違反として報告する
4. 以下の形式で報告する

---

## Output Format

違反がある場合：

```
## Comments Review 結果

### 違反あり
- **ファイル**: [ファイルパス:行番号]
- **違反内容**: 具体的な問題
- **修正方針**: 削除するか WHY に書き換える
```

違反がない場合：

```
## Comments Review 結果

チェック完了。問題なし。
```
