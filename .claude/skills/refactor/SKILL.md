# Refactor Skill

## Overview

振る舞いを変えずに内部構造を改善する。

リファクタリングの鉄則：**テストが全通過するなら振る舞いは保たれている**。

---

## Steps

### Step 1: リファクタリング目的の把握

何を改善するか・なぜ改善するかを確認する。

```
## リファクタリング目的
- 対象: [ファイルパス]
- 現在の問題: ...
- 改善後の期待状態: ...
- 振る舞いへの影響: なし（振る舞いを変えないことを確認）
```

---

### Step 2: 新規ファイル設計確認（新規ファイルがある場合のみ）

新規ファイルを作成する場合は、実装前に以下を提示してユーザーの承認を得る。

```
## 新規ファイル設計

| ファイルパス | 責務 | export するもの |
|---|---|---|
| ... | ... | ... |
```

承認を得てから次へ進む。新規ファイルがない場合はスキップする。

---

### Step 3: テスト通過確認（リファクタ前）

`npm run vitestrun` を実行し、現在の全テストが通過していることを確認する。

テストが失敗している場合は、リファクタリングを開始しない（先に bug-fix で解消する）。

---

### Step 4: リファクタリング実施

Step 1 の目的に沿って内部構造を改善する。

---

### Step 5: テスト通過確認（リファクタ後）

`npm run vitestrun` を実行し、全テストが引き続き通過することを確認する。

テストが落ちた場合はリファクタリングを見直す（振る舞いが変わっている可能性がある）。

---

### Step 6: レビュー実行

以下のレビューを実行する：

- frontend-review
- architecture-review
- comments-review

---

### Step 7: session-retrospective 実行

全ステップ完了後に session-retrospective を実行する。

---

## Constraints

- 新規ファイルを作成する場合は Step 2 でファイル名・責務・export するものを提示し、ユーザーの承認を得てから実装に進む
- Step 3・5 のテスト確認は必ず実施する。省略しない。
- リファクタリング中に仕様変更・バグ修正を混入させない
- テストが落ちた状態で「振る舞いは変わっていない」と判断しない
- リファクタリングの範囲を指示の対象に限定する（スコープクリープしない）
