# Bug Fix Skill

## Overview

バグ修正を 原因調査 → 失敗テスト作成 → 修正実装 → 確認 の順で一貫して進める。

TDD 順序（失敗するテストを先に書き、修正で通す）を崩さないことが目的。

---

## Steps

### Step 1: 原因調査

以下を確認して原因を特定する：

- どの条件で発生するか（再現条件）
- エラーメッセージ・スタックトレース
- 関連するコード（コンポーネント / hooks / api クライアント / schema）
- 直近の変更（git log で確認）

調査結果を以下の形式でまとめる：

```
## 原因調査結果

### 再現条件
- ...

### 原因箇所
- ファイル: [ファイルパス:行番号]
- 原因: ...

### 根本原因
- ...
```

---

### Step 2: 新規ファイル設計確認（新規ファイルがある場合のみ）

修正に新規ファイルの作成が伴う場合は、実装前に以下を提示してユーザーの承認を得る。

```
## 新規ファイル設計

| ファイルパス | 責務 | export するもの |
|---|---|---|
| ... | ... | ... |
```

承認を得てから次へ進む。新規ファイルがない場合はスキップする。

※ Step 3 で作成する失敗テストファイル（`.test.ts` / `.test.tsx`）はこの承認の対象外とする
（Step 3 の必須手順として別途規定されているため）。対象は実装用の新規ファイル
（新規ユーティリティ・新規コンポーネント等）に限る。

---

### Step 3: 失敗するテストを作成する

修正前の状態でテストを書き、意図的に失敗させる（Vitest / Testing Library を使用）。

- `.test.ts` / `.test.tsx` にテストを追加する

テストが失敗することを確認してから次に進む。

---

### Step 4: 修正実装

Step 3 のテストが通るように修正する。

---

### Step 5: テスト通過確認

`npm run vitestrun` でテストが通ることを確認する。

---

### Step 6: 横展開確認

horizontal-gap-check を実行し、同じバグが他のファイルにも存在していないか確認する。
存在する場合は同様に修正する。

---

### Step 7: レビュー実行

以下を**全件確認**してから次へ進む（1つでも未実行の場合は Step を完了としない）：

- [ ] frontend-review
- [ ] architecture-review
- [ ] comments-review

---

### Step 8: NG 対応

Step 7 で NG があった場合、skill-gap-detector を実行し、既存 skill の漏れを確認する。

---

### Step 9: session-retrospective 実行

全ステップ完了後に session-retrospective を実行する。

---

## Constraints

- 新規ファイルを作成する場合は Step 2 でファイル名・責務・export するものを提示し、ユーザーの承認を得てから実装に進む（Step 3 で作成するテストファイルは対象外）
- Step 3（失敗テスト作成）は必ず実施する。省略しない。
- テストが失敗していない状態で Step 4 に進まない
- 原因不明のまま修正を開始しない（Step 1 で必ず根本原因を特定する）
- バグ修正に便乗して仕様変更・リファクタリングを行わない
