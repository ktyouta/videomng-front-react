# UI Consistency Review Skill

## Overview

UI 変更の前に画面全体を俯瞰し、個別対応によるデザイン崩壊を防ぐ。

「場当たり的な修正」を避けるため、変更前に以下の観点で全コンポーネントを評価する。
提案・修正方針の出力のみ行い、ファイルの変更はこのスキルでは行わない。

このプロジェクトはスタイリング手法が **MUI・react-bootstrap/bootstrap・styled-components・emotion・semantic-ui-react** で混在しているため、
「Tailwind クラス名の統一」ではなく「同一画面・同一機能内で使われている手法・見た目のトークンとの整合性」を評価軸にする。

---

## Steps

### Step 1: 対象画面のコンポーネントを収集する

対象は「レビュー対象のモーダル・画面が属するサブディレクトリ」（例: `features/home/components/videodetail/`）であり、feature全体（例: `features/home/components/`配下すべて）ではない。Globでそのサブディレクトリ配下の全ファイルを列挙し、モーダル本体・直接の親コンポーネントだけに絞らず、列挙した全ファイルを読む。一部のファイルだけ読んで評価を始めない。

---

### Step 2: 一貫性チェック

以下の観点で全コンポーネントを横断的に確認する。

**カラーパレット**
- 使用されている色（16進数コード・MUI theme のカラーキー・Bootstrap のカラークラス）を列挙する
- 同種の要素（エラーテキスト・プライマリボタン等）に異なる色指定が混在していないか
- 背景色・ボーダー色・アクセントカラーが画面内で統一されているか

**スペーシングリズム**
- `margin` / `padding` / `gap` の値（インライン style・styled-components・MUI の `sx` prop 等）を列挙する
- 隣接セクション間の余白が不規則でないか
- カードやボタンの内側余白（padding）が揃っているか
- `gap` を使うコンテナに新規要素を追加する場合、近接させたい要素同士とセクション間隔を区別しているか（区別が必要なら入れ子コンテナでグループ化する）
- 個別要素へのmarginで間隔を調整する場合、`gap`との加算が起きないか（flexbox/gridの`gap`は個別marginを上書きせず加算される）

**タイポグラフィ階層**
- フォントサイズ・フォントウェイトの組み合わせを列挙する
- 見出し・ラベル・補助テキスト・エラーテキストの役割ごとに一貫したスタイルが使われているか

**視覚的重みのバランス**
- 各セクションの視覚的な存在感が意図通りの優先順位になっているか（主役・脇役・補助）
- 特定のコンポーネントが過剰・過小に主張していないか

**インタラクションの統一感**
- hover / active / disabled の表現が画面内で統一されているか
- transition の有無・種類が揃っているか

**スタイリング手法の混在**
- 同一画面・同一コンポーネント内で複数のスタイリング手法（例: styled-components と MUI の `sx` prop）が理由なく混在していないか
- 混在させる場合、既存の同種コンポーネントがどちらを使っているかを踏襲しているか

---

### 修正提案の書き方（重要）

- 1つの提案は1〜2文の短い文で書く。長い1文に条件や理由を詰め込まない
- 専門用語（ハードコード・非選択状態 等）を使う場合は、その場で日常語に言い換える
  （例:「コードに直接書き込まれている」「まだ選んでいない方」）
- 「何が問題か」を一言で先に書き、次に「どうすればいいか」を書く。理由の説明は最後に付け足す程度にとどめる
- 前提知識のない人（中学生程度）が読んでも意味が伝わるかを基準にする

---

### Step 3: 評価を出力する

以下の形式で出力する。問題がなければ「問題なし」と出力して終了する。

```
## UI Consistency Review

### 現状の評価

#### カラーパレット
- 使用色: [一覧]
- 気になる点: [あれば記載、なければ「問題なし」]

#### スペーシングリズム
- 使用余白: [一覧]
- 気になる点: [あれば記載、なければ「問題なし」]

#### タイポグラフィ階層
- 使用スタイル: [一覧]
- 気になる点: [あれば記載、なければ「問題なし」]

#### 視覚的重みのバランス
- 気になる点: [あれば記載、なければ「問題なし」]

#### インタラクション
- 気になる点: [あれば記載、なければ「問題なし」]

#### スタイリング手法の混在
- 気になる点: [あれば記載、なければ「問題なし」]

### 修正提案（優先度順）

1. [最優先: 最も影響範囲が大きい問題]
2. [次点]
3. [軽微]

→ 上記を踏まえた上で変更を進めます。
```

---

## Constraints

- ファイルの変更は行わない（評価・提案のみ）
- 一覧の列挙は省略しない（省略すると見落としが生じる）
- 「たぶん」「おそらく」ではなく、コード上で確認できた事実のみ記載する
- 提案は優先度順に並べ、最大5件に絞る
- 修正提案は平易な言葉で書く（専門用語の言い換え・一文一意を徹底する）
