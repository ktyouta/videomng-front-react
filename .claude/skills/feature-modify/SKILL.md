# Feature Modify Skill

## Overview

既存機能の改修を、現状把握 → 影響範囲調査 → 差分設計 → 確認 → 実装 → レビュー → 仕様突き合わせ の順で一貫して進める。

feature-impl との違い：
- 既存コードを読んでから設計する（グリーンフィールドではない）
- horizontal-scope で影響ファイルを洗い出す
- 設計は「差分設計」（何が変わるか）を中心にする

---

## Steps

### Step 0: UI 変更の場合は ui-consistency-review を先に実行する

改修が UI（コンポーネント・レイアウト・スタイル）を含む場合、Step 1 に進む前に ui-consistency-review を実行する。

以下をすべて含む（既存コンポーネントへの要素追加・削除もここに含まれる）：
- 新規コンポーネントの追加
- 既存コンポーネントへのボタン・入力・セクション等の追加・削除
- レイアウト・スタイルの変更

ui-consistency-review の結果を踏まえた上で、Step 1 の設計・実装に入る。

UI 変更を含まない場合はこの Step をスキップする。

---

### Step 1: 改修内容の把握

以下を確認する：

- 何を変えるか（変更前の動作・変更後の期待動作）
- なぜ変えるか（仕様変更・要望・問題解決）
- どのコンポーネント・hooks・API 連携に影響するか

---

### Step 2: 既存実装の確認

以下を読んで現状を把握する：

- `docs/[機能名]/plan.md`（設計書が存在する場合）
- `docs/[機能名]/spec.md`（仕様書。存在する場合は読み込み、現行仕様を把握する。存在しない場合は「無いのでスキップ」とせず、Step 4 で既存の挙動＋今回の変更内容を仕様書として新規作成する）
- 改修対象の既存コード（コンポーネント / hooks / API 呼び出し）

**バグ修正を含む改修の場合は追加で以下を行う：**
- 問題が発生しているイベントフロー・データフロー・呼び出し経路を図示またはコメントで明示する
- 「どこで何が起きているか」を経路単位で追跡してから修正方針を決める
- 修正前に「この修正で経路上のどこが変わるか」を確認する
  - 曖昧なまま修正すると「直っていない」が複数サイクル発生する原因になる

**途中まで実装された既存コードを踏襲・複製する場合は追加で以下を行う：**
- そのロジックが意図通り動作するか（非同期データの反映漏れ、useState同士の派生関係の整合性等）を検証してから踏襲する
- 動作未検証のまま複製すると、既存コードに潜むバグも一緒に複製してしまう

---

### Step 3: horizontal-scope で影響範囲を洗い出す

horizontal-scope を実行し、今回の変更パターンと同じ対応が必要なファイルを特定する。

洗い出した対象ファイルへの変更ロジックが完全に同一になる場合、Step 4 で個別実装ではなく共通化を優先的に検討する。

「小規模な変更だから」「対象ファイルが少ないから」という理由でこの Step 自体を省略しない。省略すると水平展開すべきファイルの見落としに直結する。

---

### Step 4: 差分設計を出力

変更前・変更後の差分を中心に設計をまとめる。

**出力の書き方（一度に詰め込まない）**
- 詳細な表の前に、変更点を3〜5個程度の平易な箇条書きで先にまとめる（技術的な一覧はその後に続ける）
- 1項目は1〜2文で書く。長い一文に複数の理由・条件を詰め込まない
- 専門用語を使う場合はその場で言い換える（ui-consistency-review と同じ基準）

```
## 差分設計

### 要点（何がどう変わるか）
- [平易な言葉で3〜5個の箇条書き]

### 変更内容
- [変更前]: ...
- [変更後]: ...

### 変更点
- コンポーネント: ...
- hooks: ...
- API 連携: ...

### 変更・影響ファイル一覧

| ファイルパス | 種別 | 操作 |
|---|---|---|
| ... | ... | 変更 / 新規 / 削除 |

### 仕様要件チェック（docs/[機能名]/spec.md 準拠）
docs/[機能名]/spec.md が存在する場合、この差分設計を踏まえて更新する。存在しない場合は「無いのでスキップ」とせず、既存の挙動＋今回の変更内容を要件項目としてこの場で新規作成する（後続の spec-review が照合できる箇条書き・チェックリスト形式）。

- spec.md から要件項目を抽出したか
- 全ての要件項目が差分設計（コンポーネント・hooks・API）に反映されているか

### plan.md 更新
docs/[機能名]/plan.md が存在する場合、今回の改修内容をタスクとして追記する。存在しない場合は「無いのでスキップ」とせず、この差分設計を元に新規作成する（feature-resume が今回の改修を再開ポイントとして拾えるようにするため）。

### フォルダ構成チェック（CLAUDE.md 準拠）
- `features/<機能名>/` の既存配置規則（components/ hooks/ api/ types/ schemas/）に沿っているか
- クエリキー（`queryKey.ts` 等）の追加・変更が既存定義に集約されているか

### 共通化チェック
- 同一の変更ロジックを複数ファイルへ適用する場合、個別実装ではなく共通コンポーネント/hooksへの切り出しを検討したか
- 個別実装を選ぶ場合、その理由（構造が微妙に異なる等）を明記する

問題がなければ実装に進みます。よろしいですか？
```

ユーザーの OK を得てから次へ進む。

---

### Step 5: 実装

Step 4 の差分設計に沿って変更する。

変更に入る前に以下を必ず行う：

**5-a. Step 0 未実施チェック**

以下のいずれかに該当する場合は「UI 変更あり」と判定する：
- HTML 要素の追加・削除
- スタイル（className・styled-components・MUI の `sx` prop 等）の変更
- コンポーネントの追加・削除・入れ替え

「UI 変更あり」かつ Step 0 が未実施の場合、ui-consistency-review を今すぐ実行してから実装に進む。

**5-b. 対象画面の関連ファイルをすべて Read する**
- 対象画面のコンポーネント・ページファイルだけでなく、レイアウト・親コンポーネントも含む全ファイルを確認する
- 「雰囲気だけ把握した」ではなく「全ファイルを読んだ」状態で変更に入る
- クリッカブルカードなどのインタラクティブ要素を変更する場合、HTML 要素の選択（`div` vs `button` 等）も既存コンポーネントを Read して踏襲する

**5-c. 変更完了後に型エラーがないか確認する**

`npx tsc --noEmit` を実行し、型エラーが 0 件になってから Step 6 に進む。

---

### Step 6: レビュー実行

以下を**全件実行**してから次へ進む（1つでも未実行の場合は Step を完了としない）：

- [ ] frontend-review
- [ ] architecture-review
- [ ] comments-review
- [ ] performance-check

レビュー完了後、会話中に（Step 0〜6のいずれかの時点で）ユーザーからの修正・指摘発話（typo・細かい文言調整を除く）があった場合、`.claude/incidents.md` への記録要否をユーザーに確認する：「今回の指摘を #feature-modify のインシデントとして記録しますか？（記録内容: [指摘の要約を1行で]）」

---

### Step 7: 仕様突き合わせ・設計書更新

Step 4 で spec.md を作成・更新済みのため、常に spec-review を実行する。

spec-review 完了後、`docs/[機能名]/plan.md` の該当タスクを `[x]` に更新する（Step 4 で新規作成した場合は今回のタスクを完了扱いにする）。NG が残っている場合は `[x?]`（仮マーク）にとどめる。

---

### Step 8: NG 対応

Step 6・7 でいずれかの NG があった場合、**その場で直接修正して解消できた場合も含めて**、
skill-gap-detector を実行し、既存 skill の漏れを特定して修正提案を行う。
「軽微だから」「即座に直せたから」という理由での省略は認めない。

---

### Step 9: session-retrospective 実行

全ステップ完了後に session-retrospective を実行する。

---

## Constraints

- 各 Step は順番通りに実行する（並行実行しない）
- `docs/[機能名]/spec.md` / `docs/[機能名]/plan.md` が存在しない場合、「無いのでスキップ」で済ませず Step 4 で新規作成する（省略しない）
- ユーザー確認（Step 4）では必ずユーザーの明示的な OK を得てから次に進む
- 既存テストが存在する場合は、変更後も通過するか確認する（`npm run vitestrun`）
- 指示にない機能を変更に追加しない
- エラーや NG を無視して次のステップに進まない
- コーディング規約・チェック項目の追加は skill ファイルにのみ行う。skill で対応できる内容を CLAUDE.md に追記しない
