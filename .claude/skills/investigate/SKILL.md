# Investigate Skill

## Overview

調査・確認・原因究明を依頼されたとき、関連ファイルを網羅的に読み込んでから回答する。

「起点ファイルだけ読んで回答する」という局所的な調査を防ぐことが目的。
Phase 1 で起点を特定し、Phase 2 でファイルをたどり切ってから回答する。

---

## Steps

### Phase 1: 起点探索

調査対象に関連する起点ファイルを特定する。

1. ユーザーの発話から調査対象（ファイル名・機能名・コンポーネント名・API・エラー文言）を確認する
2. Glob で対象フォルダのディレクトリ構造を確認する
3. Grep で関連シンボル・キーワードの出現ファイルを特定する
4. 起点ファイルのリストを作成する（この時点ではまだ Read しない）

> **注意**: ユーザーが具体的なファイル名・行番号を示さずに症状（見た目・挙動の不具合）だけを報告した場合、直近の会話で話題になっていたコードを根拠なく起点と決め打ちしない。該当しそうな箇所が複数候補あり得る場合は、Grep等で候補を洗い出してから絞り込む。

---

### Phase 2: 関連ファイル展開

起点ファイルから関連ファイルをたどり、全件読む。

1. 起点ファイルを全件 Read する
2. 読んだファイルの **import 文・参照先・型定義ファイル** を確認する
3. 新たに見つかった関連ファイルを Read する
4. 新たなファイルが見つからなくなるまで 2〜3 を繰り返す

> **スキップ条件**: import 先が `node_modules` 配下の場合はスキップしてよい。ただし調査対象がライブラリ自体の内部実装・スタイル生成ロジックである場合はスキップせず読み込む

---

### Phase 3: 回答

全ての関連ファイルを読んだ状態で回答する。

回答の冒頭に読み込んだファイルの一覧を記載する：

```
## 読み込んだファイル
- path/to/file-a.ts
- path/to/file-b.ts
...
```

---

## Constraints

- Phase 2 を省略しない。「起点ファイルだけ読んで回答した」状態はこのスキルの違反
- 「ファイルが多すぎる」という理由で Phase 2 をスキップしない
- 回答冒頭の「読み込んだファイル一覧」を省略しない
- 関連ファイルを全件読み終える前に回答を開始しない
- CSS・ライブラリの挙動確認のために、dev server・ブラウザ（ヘッドレス含む）・使い捨ての検証用コンポーネント/ページを自分の判断で起動・作成しない。確認は自プロジェクトのソースと `node_modules` 内のライブラリソースを読み、CSSプロパティの値を実際に追跡・計算する形で行う
- ソースから値が確定できる場合、「おそらく」「〜と思われる」のような推測表現を使わず、根拠となるファイル・行を示した上で断定する
