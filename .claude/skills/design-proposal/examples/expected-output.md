# Expected Output — 設計案提示

## 入力例

「お気に入り動画の削除を論理削除にしたい。どう設計すればいい？」

---

## 期待される出力

```
## 設計案

### 案A: isDeleted フラグを追加する

お気に入りのレスポンス・ローカル状態に `isDeleted: boolean` を持たせ、
削除時にサーバー側で true に更新し、一覧取得時にフィルタする。

**メリット**
- 実装がシンプル
- 復元が容易（フラグを false に戻すだけ）

**デメリット**
- 一覧取得のたびにフィルタ条件が必要になる
- フィルタの書き忘れによるバグリスクがある

**実装コスト**: 低

---

### 案B: deletedAt を追加する

`isDeleted` の代わりに `deletedAt: string | null` を追加する。

**メリット**
- 削除日時が記録できる
- 案Aと同じ実装コストで情報量が増える

**デメリット**
- 画面側で使うのは真偽判定のみのため、フラグより扱いがわずかに増える

**実装コスト**: 低

---

### 案C: 削除済み一覧を別APIに分離する

一覧取得APIとは別に「削除済みお気に入り一覧」専用のエンドポイントを用意する。

**メリット**
- 通常一覧のレスポンスがシンプルなまま保てる

**デメリット**
- API呼び出し・react-queryのキャッシュキー管理が増える
- 通常の一覧と削除済み一覧の整合性を保つ実装が必要

**実装コスト**: 高

---

## 推奨案: 案B（deletedAt 追加）

**理由**: 実装コストは案Aと同等でありながら削除日時も記録でき、将来「〇日前に削除」のような表示要件が出ても対応しやすい。

## 破綻するケース

- 削除済みデータを別画面で大量に一覧表示する要件が出た場合、通常一覧のレスポンスに削除済み分が混在しないよう案Cへの移行を検討する必要がある
- 削除件数が膨大になり一覧取得のレスポンスサイズが問題になった場合、ページング・専用APIの設計見直しが必要になる
```
