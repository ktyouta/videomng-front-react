# Horizontal Scope Skill

## Overview

実装前に「今回の変更パターンと同じ対応が必要な他のファイル」を洗い出し、
一覧で提示する。

実装漏れ・水平展開の抜けを防ぐことが目的。

---

## Instructions

### Step 1: 変更パターンを特定する

今回の変更が「どの種類の変更か」を把握する。

代表的なパターン例：
- 新ページ・新画面追加
- 新コンポーネント追加
- 既存コンポーネントへの新 Props・新フィールド追加
- API 呼び出し・レスポンス型の変更
- 共通型定義の変更

---

### Step 2: 水平展開対象を洗い出す

以下の観点でコードベースを調査し、対象ファイルを特定する。

#### フロントエンド観点
- 同種のコンポーネント・ページ（`features/<機能名>/components/`）
- 同種の hooks（`use*.ts`）
- ルート定義（`react-router-dom` のルーティング設定）
- ナビゲーション・ヘッダー・フッター・サイドメニュー
- API 呼び出し箇所（`features/<機能名>/api/`、`features/api/`、`lib/apiClient.ts`）
- react-query のクエリキー定義（`queryKey.ts` 等）
- テストファイル（`*.test.ts` / `*.test.tsx`）

#### 共通観点
- 型定義ファイル（`types/`、`schemas/`）
- 定数・設定ファイル（`consts/`、`const/`）
- グローバルストア（`store/` の jotai atom）

#### 対象外と判定する場合の確認事項
- 候補ファイルを「同じパターンではない」として除外する場合、画面内の文言・見出しだけで判断しない
- 対象コンポーネントの呼び出し元（親コンポーネント）を確認し、「どのボタン・どの条件から呼び出されるか」（新規登録／編集／閲覧専用等のフロー種別）を確認してから除外を判定する
- 呼び出し元の確認なしに「文言が似ているから別フロー」「文言が違うから同一フロー」と判断しない

---

### Step 3: 結果を出力する

examples/expected-output.md の形式で一覧提示する。

---

## Constraints

- 実装は行わない（洗い出しのみ）
- 「おそらく不要」なファイルは含めない（確信があるものだけ列挙する）
- ファイルが存在しない場合は「新規作成が必要」と明記する
